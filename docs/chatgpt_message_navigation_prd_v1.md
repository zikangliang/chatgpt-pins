- **文档版本**：v1.0

- **文档日期**：2026-03-19

## 1. 项目概述

### 1.1 项目背景
在 ChatGPT 的长会话场景中，用户经常需要回看自己此前发送过的某一条消息，例如需求描述、代码片段、关键指令、问题分支或中间结论。当前页面主要依赖线性滚动浏览历史内容，当单个会话消息数量较多时，用户需要反复上滑查找，定位成本高，且容易丢失上下文。

页面右侧现有滚动条仅提供粗粒度滚动能力，并不具备基于“消息节点”的精准跳转功能。因此，有必要设计一款轻量级导航增强插件，帮助用户在超长对话中快速定位和回跳历史消息。

### 1.2 项目目标
开发一款运行于 ChatGPT Web 端的浏览器插件，自动识别当前会话中的用户消息，在页面右侧生成可点击的导航锚点，支持悬停预览、平滑跳转、目标高亮和当前位置同步，以降低长会话回溯成本并提升信息检索效率。

### 1.3 一句话定义
为 ChatGPT 长会话提供“按消息节点跳转”的目录导航能力。

---

## 2. 用户与场景

### 2.1 目标用户
本产品主要面向以下用户群体：

1. **重度使用 ChatGPT 的用户**  
   经常进行多轮对话，单个会话较长。

2. **以任务驱动为主的用户**  
   会在同一会话中持续补充需求、追问细节、修改指令。

3. **技术/内容/产品类用户**  
   需要频繁回看历史输入，例如代码、文案、分析框架、产品需求等。

### 2.2 核心使用场景
#### 场景 A：回看自己之前的某次提问
用户与 ChatGPT 已进行几十轮以上交流，想快速回到此前某一条关键提问，而不是连续滚动翻找。

#### 场景 B：在多个问题分支间来回切换
用户在同一会话中先后讨论多个主题，需要快速回到某一分支的起点。

#### 场景 C：长需求迭代中的上下文对照
用户曾在前文中给出详细约束或原始需求，希望快速跳回该位置进行对照。

#### 场景 D：调试与代码问答
用户多次粘贴代码、报错信息、修复思路，希望快速定位到某一段自己提交的输入。

## 3. 需求定义

### 3.1 核心问题
当前 ChatGPT 长会话存在以下问题：

1. **历史消息定位困难**  
   用户无法快速找到自己在前文中的特定输入内容。

2. **会话结构不可见**  
   页面呈现为长线性流，没有目录、锚点或可视化导航结构。

3. **缺乏基于消息的精准跳转能力**  
   原生滚动只能粗略浏览，无法按消息节点直接定位。

### 3.2 产品价值
本产品的价值体现在以下方面：

- 缩短用户回看历史输入的时间
- 降低长会话中的滚动操作成本
- 提升对话结构可读性
- 让长会话具备近似“目录导航”的浏览体验
- 为后续扩展功能（搜索、收藏、标记、摘要）提供基础索引能力

## 4. 产品范围

### 4.1 第一版产品范围（In Scope）
第一版仅聚焦于当前会话内的导航增强能力，包含以下范围：

1. 识别当前会话中的用户消息
2. 基于用户消息生成导航锚点
3. 在页面右侧展示消息导航栏
4. 支持悬停预览消息摘要
5. 支持点击后平滑跳转到目标消息
6. 支持跳转后目标消息高亮
7. 支持用户滚动时同步当前激活项
8. 支持消息动态新增后的索引刷新
9. 支持会话切换后的导航重建
10. 适配深色/浅色模式的基本显示

### 4.2 第一版不包含（Out of Scope）
以下内容不纳入第一版：

1. 跨会话索引与全局目录
2. 云端同步
3. 移动端适配
4. AI 自动总结消息标题
5. 消息收藏、导出、标签管理
6. 对 ChatGPT 页面进行大面积布局改造
7. 复杂全文搜索
8. 多账户、多工作区同步管理

### 4.3 平台范围
第一版建议仅支持以下环境：

- ChatGPT Web 端
- Chrome 浏览器
- Edge 浏览器
- 桌面端优先

## 7. 功能需求

## 7.1 功能一：消息识别与索引

### 功能说明
插件自动识别当前会话页面中的用户消息节点，并生成结构化索引数据，用于导航栏渲染与跳转。

### 输入
- 当前页面已渲染的会话消息 DOM

### 输出字段建议
- 消息唯一标识 `id`
- 顺序编号 `index`
- 消息角色 `role`
- 原始文本 `text`
- 摘要文本 `summary`
- DOM 元素引用 `element`
- 页面位置 `topOffset`
- 可选时间字段 `timestamp`

### 规则要求
1. 默认仅索引“用户消息”
2. 需正确区分用户消息与助手回复
3. 对富文本、多段消息、代码块消息做兼容处理
4. 支持后续扩展为“仅助手”或“全部消息”模式

### 验收标准
- 正常识别绝大多数用户消息
- 会话切换后重新建立索引
- 不出现大量重复索引或漏索引

---

## 7.2 功能二：右侧导航栏展示

### 功能说明
在页面右侧滚动区域附近渲染一个悬浮导航栏，展示当前会话中的消息锚点。

### 展示要求
1. 默认以低干扰方式显示
2. 支持折叠 / 展开
3. 默认按照消息发送顺序排列
4. 对当前激活项进行高亮展示

### 默认展示模式建议
第一版建议采用“编号 + 摘要截断”混合模式：

- `[01] 请帮我整理需求`
- `[02] 我想开发一个插件`
- `[03] 右侧滚动跳转功能`

### 验收标准
- 不明显遮挡主内容区
- 不与浏览器原生滚动条严重冲突
- 展开与收起行为稳定

---

## 7.3 功能三：悬停预览

### 功能说明
用户将鼠标悬停在导航项上时，展示该消息的摘要信息，帮助用户判断是否为目标内容。

### 预览内容建议
- 第几条用户消息
- 消息摘要（前 30～50 字）
- 可选位置信息（如“会话约 40% 处”）

### 交互要求
1. Tooltip 出现位置应避免被屏幕边缘裁切
2. 文本过长时自动截断
3. 不应频繁闪烁或抖动

### 验收标准
- 悬停反馈清晰
- 预览内容与对应消息一致
- 长会话下预览响应正常

---

## 7.4 功能四：点击跳转

### 功能说明
用户点击导航项后，页面滚动至对应消息位置，并提供明确视觉反馈。

### 跳转行为要求
1. 使用平滑滚动
2. 定位到目标消息附近
3. 考虑顶部偏移，避免消息被吸附区域遮挡
4. 跳转完成后对目标消息进行短暂高亮提示

### 高亮要求
1. 高亮应足够明显
2. 不改动原消息内容
3. 高亮持续时间可控制在 1.5～2 秒左右

### 验收标准
- 点击后可稳定跳转到正确消息
- 跳转误差小
- 高亮效果明显但不过度打扰

---

## 7.5 功能五：当前阅读位置同步

### 功能说明
用户手动滚动会话时，插件自动判断当前视口最接近的用户消息，并同步高亮对应导航项。

### 规则建议
1. 以视口顶部或中上区域最近消息为激活项
2. 快速滚动时高亮切换应平稳
3. 不应出现频繁跳闪或滞后过大

### 验收标准
- 当前激活项能较准确反映阅读位置
- 长会话滚动时性能稳定

---

## 7.6 功能六：动态更新

### 功能说明
由于 ChatGPT 页面是动态渲染的，插件需要在消息新增、重渲染、会话切换等场景下自动更新导航索引。

### 更新场景
1. 用户发送新消息
2. 会话内容延迟渲染
3. 页面局部重绘
4. 用户切换到其他会话

### 验收标准
- 新消息出现后，导航列表自动更新
- 切换会话后旧导航数据被清理
- 不因频繁变动导致重复重建或性能下降

## 8. 界面与交互要求

## 8.1 导航栏位置
建议默认位于页面右侧中部偏上位置，靠近滚动条但与其保留安全间距。

## 8.2 导航栏状态
支持以下两种状态：

### 折叠状态
- 占用空间最小
- 仅展示简短入口或极简锚点

### 展开状态
- 展示完整导航项列表
- 可滚动浏览全部消息锚点

## 8.3 导航项样式要求
每一项建议包含：
- 编号
- 摘要文字
- 激活态样式
- Hover 态样式

## 8.4 主题适配
需兼容 ChatGPT 的深色与浅色模式：
- 文本可读
- 激活项对比度足够
- Tooltip 与面板边框清晰

## 8.5 交互原则
1. 低干扰，不破坏原页面使用体验
2. 操作明确，反馈及时
3. 长会话下依然保持可用性
4. 样式风格尽量贴近页面原有设计语言

---

## 9. 非功能需求

### 9.1 性能要求
1. 插件不应显著降低页面滚动流畅度
2. 导航更新应尽量节流，避免频繁重建
3. 长会话中应避免明显卡顿

### 9.2 稳定性要求
1. 页面刷新后插件可自动恢复
2. 会话切换后可正常重建导航
3. 动态消息渲染场景下不应频繁失效

### 9.3 兼容性要求
1. 支持 Chrome 最新稳定版
2. 支持 Edge 最新稳定版
3. 优先适配桌面宽屏环境

### 9.4 可维护性要求
1. 尽量减少对易变 DOM class 的强依赖
2. 选择器与页面识别逻辑应模块化封装
3. 为后续扩展“搜索、收藏、助手消息模式”保留结构空间

## 10. 技术约束与实现建议

> 本节用于给开发团队提供实现方向，不替代详细技术设计文档。

### 10.1 推荐实现形态
建议使用 **浏览器扩展（Manifest V3）** 实现。

### 10.2 推荐模块划分
1. 页面识别模块
2. 消息抽取模块
3. 索引管理模块
4. 导航栏 UI 模块
5. 跳转与高亮模块
6. 滚动同步模块
7. 动态监听模块
8. 设置与配置模块（可后置）

### 10.3 关键实现建议
1. 使用 `MutationObserver` 监听消息变化
2. 使用 `IntersectionObserver` 或节流滚动逻辑同步当前激活项
3. 将消息索引结构化缓存，减少重复计算
4. Tooltip 与高亮样式采用独立 class 管理
5. 保持内容脚本与页面 DOM 操作的边界清晰

## 11. 边界情况与异常场景

以下情况需在设计与测试中重点关注：

1. **极长消息**  
   摘要过长时应自动截断，避免撑坏导航栏。

2. **代码块消息**  
   应能正确提取或回退显示简化摘要，如“代码片段”。

3. **富文本或多段内容**  
   应优先抽取首段文本作为摘要。

4. **仅图片或特殊内容消息**  
   若无文本，可显示“非文本消息”或类似占位文案。

5. **会话动态加载不完整**  
   若页面尚未完全渲染，应延迟重建索引。

6. **页面结构变化**  
   如 ChatGPT 页面 DOM 结构改版，插件可能需要调整选择器策略。

## 12. 风险与应对

### 风险一：目标页面 DOM 结构变化
**说明：** ChatGPT Web 页面结构存在变更可能，依赖过深会导致插件失效。  
**应对：**
- 使用更稳健的识别策略
- 封装选择器层
- 预留兼容降级方案

### 风险二：长会话性能压力
**说明：** 会话消息过多时，滚动监听和重建索引可能造成卡顿。  
**应对：**
- 节流与批处理
- 懒更新
- 控制导航 UI 渲染量

### 风险三：UI 与页面原元素冲突
**说明：** 右侧悬浮导航可能与原滚动条或其他浮层冲突。  
**应对：**
- 允许折叠
- 控制默认宽度
- 保留边距并支持后续位置配置

### 风险四：动态渲染导致状态不一致
**说明：** 新消息生成、会话切换和局部刷新可能导致索引过期。  
**应对：**
- 监听关键节点变化
- 统一索引重建入口
- 做旧状态清理

## 17. 附录：项目描述（可用于 README / 立项说明）

开发一款面向 ChatGPT Web 端的浏览器插件，用于增强长会话场景下的历史消息导航能力。插件自动识别当前会话中用户发送的消息，并在页面右侧生成可点击的导航锚点，支持悬停预览、平滑跳转、目标高亮和当前位置同步，从而显著降低长对话回溯成本并提升浏览效率。
